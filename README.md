# Agentic Tool-Use Agent for CI/CD Failure Diagnosis

This project builds an **agentic tool-use agent** for diagnosing CI/CD and ML-engineering failures.

The goal is to train an agent that decides:

```text
What tool should I use next?
When do I have enough evidence to answer?
```

The agent works over a fixed action space:

```text
retrieve_logs
inspect_repo
search_docs
run_static_check
final_answer
```

At test time, the system can optionally use **web search** to enrich the final answer, but web search is intentionally kept **outside training** so the environment stays deterministic.

## 1. Project Idea

Each failed run is one diagnosis case. The policy receives a user query such as:

```text
My GitHub Actions workflow failed. Diagnose the root cause and suggest a fix.
```

Instead of answering immediately, the agent must choose a sequence of actions.

Example:

```text
Good:
retrieve_logs -> inspect_repo -> search_docs -> final_answer

Bad:
search_docs -> final_answer
```

The final answer is currently generated by a template-based answer generator using collected evidence. The learned part of the system is mainly the **tool-use policy**.

## 2. Current Pipeline vs Standard RLHF

The project is easiest to understand through the familiar RLHF pipeline:

### Standard RLHF Pipeline

```text
Step 1: Supervised Fine-Tuning (SFT)
Step 2: Reward Model Training
Step 3: PPO / RL Optimization
```

### This Project's Pipeline

```text
Step 1: Behavior Cloning (SFT-like warm start)
Step 2: Reward Definition
Step 3: RL Optimization
Step 4: Optional Preference Optimization later
```

Notes:

- PPO is the **RL optimizer**, not the environment itself.
- RLHF still needs an environment or rollout loop, but its distinctive part is usually the **learned reward model** plus PPO-style optimization.
- In this project, we currently do **not** train a reward model.
- Instead, we define reward directly inside the environment with hand-designed scoring based on:
  - correct failure type
  - correct diagnosis
  - correct fix
  - evidence use
  - tool efficiency

So the closest mapping is:

```text
SFT                -> Behavior Cloning from expert_trajectories.jsonl
Reward Model       -> Replaced here by engineered reward logic inside env.py
PPO / RL phase     -> Replaced here by a small DQN-style optimizer in rl.py
Optional DPO stage -> Direct Preference Optimization
```

So this project is better described as:

```text
BC + engineered-reward RL + optional later preference optimization
```

It is RLHF-inspired, but not standard RLHF.



### Which policy is active?

Different stages use different policy objects:

- **BC training stage**: the learned policy is a `BehaviorCloningPolicy`
- **RL training stage**: the learned policy is an `RLFineTunedPolicy`
- **DPO training stage**: the learned policy is a `DPOPolicy`
- **Inference/evaluation stage**: the code loads one saved checkpoint and uses that as the active policy

So when someone asks "are we using BC model or RL Q-network?", the answer is:

- during `train_bc.py`, the model is the BC model
- during `train_rl.py`, the model is the RL Q-network
- during `train_dpo.py`, the model is a BC-style policy further fine-tuned by DPO
- during inference, it depends on which checkpoint path you pass in

In other words, the framework does **not** use all policies at once. It moves from one stage to the next.
